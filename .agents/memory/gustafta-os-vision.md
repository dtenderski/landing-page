# Gustafta AI Operating System — arah yang disepakati

Sumber: sintesis 3 dokumen (diagnosis kode + roadmap ChatGPT + kurikulum Notion). Semua sepakat: **masalahnya kurasi, bukan kekurangan produk** (~70% fondasi sudah ada). Pemilik solo, dibangun dengan Replit.

## Model produk: 5 PILAR (framing wajah produk)
1. **COACH** — chatbot pendamping per domain (tanpa orchestrator)
2. **CLAW** — tim agen spesialis multi-agent (MultiClaw/OpenClaw)
3. **WORKROOM** — ruang kerja manusia+agen: workflow + Human Gate ◆ + log + deliverable pack
4. **ACADEMY** — kursus berjenjang Starter→Professional→Advanced→Enterprise
5. **COMPETENCY & MONETIZATION** — badge/sertifikat, portofolio, langganan, afiliasi, voucher, analytics

## Pipeline standar produk: Dialog → Blueprint → Chatbot → Workroom (arah pemilik, Jul 2026)
Pola generik yang dipakai ulang lintas domain (bukan fitur satu bidang):
1. **Dialog** — user bercerita (materi, kesulitan, kebutuhan) → Dialog Gustafta.
2. **Blueprint** — hasil dialog jadi peta tingkat pengetahuan/pengalaman + gap yang harus diperbaiki → Blueprint Engine (perlu mode "asesmen diri").
3. **Chatbot** — lahir teman berpikir & belajar yang paham gap user → builder eksisting.
4. **Workroom** — ruang uji/simulasi per domain, keluaran = **tingkat kesiapan** (skor rekomendasi).
5. **◆ Manusia memutuskan** — lanjut (ikut uji/eksekusi) atau belajar lagi; sistem tak pernah memutus sendiri.

Contoh pertama yang diminati pemilik: **uji kompetensi** (SKK/sertifikasi). Pola sama berlaku utk tender, perizinan, rekrutmen, dan "pipeline-mainframe-workflow bisnis" lain — hanya isinya yang berganti, kerangka tetap.
**How to apply**: saat membangun jalur baru bergaya klinik/pendampingan, JANGAN bikin alur ad-hoc — susun sebagai 4 tahap ini dan sambungkan ke komponen eksisting (Dialog, Blueprint Engine, builder, Workroom registry `WORKROOM_DOMAINS`).

## Aturan arsitektur (non-negotiable)
- **Modular MONOLITH, bukan microservices.** Solo dev tidak boleh punya 8 layanan + 8 DB terpisah. Batas modul = logis (DDD), satu app + satu DB.
- **Petakan konsep ke tabel yang SUDAH ADA — JANGAN bikin tabel kembar.** Data model Notion (courses/deliverables/affiliates/vouchers) menduplikasi tabel eksisting; itu keliru.
- **Adaptasi tanpa menghancurkan** yang sudah jalan (billing, 934 agen, partner, marketplace).
- **Human Gate ◆ wajib** di setiap workflow penting (uang/kontrak/keselamatan/sertifikasi/submit final).

## Pemetaan 5 Pilar → tabel eksisting (per audit)
- COACH + CLAW → `agents` (`isOrchestrator`+`agenticSubAgents` membedakan coach vs claw), `knowledge_bases`/`knowledge_chunks`/`knowledge_taxonomy`
- WORKROOM → benih di `project_brain_templates`/`project_brain_instances` (Otak Proyek) + `agentic_deliverables`. **GAP nyata**: belum ada tabel `workrooms`, `workflow_stages`, `human_gates`, `workroom_logs` (decision/assumption/risk/change) → INI yang baru dibuat saat Fase 1.
- ACADEMY → mesin SUDAH ADA: `lms_courses`/`lms_lessons`/`lms_enrollments`/`lms_lesson_progress` + `digital_certificates`/`shared_certificates`. Tinggal isi kurikulum + sambung capstone→Builder.
- COMPETENCY & MONETIZATION → `affiliates`/`mlm_commissions`, `vouchers`/`voucher_redemptions`, `certificates`, `analytics`, `store_products`/`store_orders`, `subscriptions_new`, `wa_broadcasts`. **GAP kecil**: badge + portofolio.

## Roadmap bertahap (disepakati)
- **Fase 0 — OS Shell / Kurasi** ✅ SELESAI: navigator 5 pilar di route `/os` (`client/src/pages/gustafta-os.tsx`), additive/frontend-only, entry card di dashboard "Aksi Cepat".
- **Fase 1 — Workroom MVP** ✅ SELESAI: tabel BARU `workrooms`/`workroom_gates`/`workroom_logs` (bukan reuse — benih Otak Proyek/deliverables tak cocok untuk stage-machine). Domain pertama = TENDER. Stage machine (6 tahap) + Human Gate ◆ + jurnal log (decision/assumption/risk/change/deliverable) + analisis AI GPT-4o (kelayakan+win-probability, JSON, honest [ASUMSI]). Routes `/api/workrooms*` (semua isAuthenticated + `getOwnedWorkroom` ownership guard = anti-IDOR). UI: `/workroom` (list+create) & `/workroom/:id` (detail). Deliverable disimpan sbg log type "deliverable" (meta.result), BUKAN tabel agentic_deliverables (keyed by agentId, tak cocok).
  - **Why tabel baru, bukan reuse**: agentic_deliverables & project_brain_* terikat agentId, sedangkan workroom milik user & butuh stage/gate/log terpisah. Sesuai aturan "tabel baru HANYA untuk yang belum ada".
- **Fase 2 — Aktifkan Academy** ✅ SELESAI: mesin `lms_*` + 5 kursus SUDAH ADA (seed `server/routes-lms.ts`, route `/lms`, `/lms/course/:id`) — jadi "aktivasi" = SAMBUNGKAN Academy ke Workroom (loop belajar→praktik), BUKAN bikin konten baru. Implementasi = **capstone bridge** (frontend-only, tanpa perubahan skema): kursus `category==="konstruksi"` (SBU/SKK/K3, id 3-5) menampilkan kartu Capstone → tombol buat Workroom Tender via `POST /api/workrooms` yg SUDAH ADA dengan `context={source:"capstone",courseId,courseTitle}` → navigate ke `/workroom/:id`. Workroom detail tampilkan banner backlink ke `/lms/course/:courseId` bila `context.source==="capstone"`. Mapping capstone→domain di const `CAPSTONE_BY_CATEGORY` (lms-course.tsx); saat ini hanya konstruksi→tender krn itu satu-satunya domain Workroom yg ada.
  - **Why capstone via context, bukan tabel/kolom baru**: reuse endpoint & tabel Workroom Fase 1; `context` jsonb sudah fleksibel. Aturan "tabel baru HANYA yg belum ada" → tak ada yg baru dibutuhkan.
  - **Gotcha**: halaman yg perlu bedakan 404 vs error WAJIB pakai `queryFn` custom (cek `res.status===404` → return null); default queryFn throw di semua non-2xx → `isError` selalu true, "not found" tak pernah tampil.
- **Fase 3 — Competency (badge + portofolio)** ✅ SELESAI: agregasi portofolio pribadi, TANPA tabel baru (aturan reuse). Endpoint `GET /api/portfolio` (isAuthenticated, userId-scoped, anti-IDOR — tak ada target ID user-controlled) merangkum: enrolled/completed dari `lms_enrollments`+`lms_courses`, `getWorkrooms(userId)`, capstoneCount (context.source), deliverableCount (`workroom_logs` type='deliverable', query builder `inArray` bukan raw), sertifikat `digital_certificates` active. `badges[]` diturunkan server-side (pelajar/lulusan/praktisi/capstone/produktif/tersertifikasi). UI: `client/src/pages/portofolio.tsx` (route `/portofolio`, dark theme, loading/error/empty state) + kartu tujuan pertama di pilar competency `gustafta-os.tsx`. Code review PASS.

**Why:** roadmap ChatGPT ditulis seolah greenfield & menyarankan pecah modul+DB sendiri — berisiko menghancurkan platform hidup. Notion jauh lebih baik krn berpijak aset nyata. Kesepakatan pemilik: adaptasi bertahap, kurasi dulu.

## Lanjutan (opsi pasca-Fase 3, dikerjakan "berurutan saja")
- **Opsi #1 — Perluas Workroom ke domain baru (dulu tender-only)** ✅ SELESAI: registry `WORKROOM_DOMAINS` (tender/perizinan/skk/k3) di `server/routes.ts` menggantikan hardcoded TENDER_STAGES — tiap domain punya {label, stages[], analyzeContent, systemPrompt}. Create route coerce domain→known key + pakai stages per-domain; analyze route pakai `cfg.systemPrompt` per-domain + brief builder generik (iterasi ctx via WORKROOM_CTX_LABELS) + log/meta domain-aware. **Bentuk JSON output analisis TETAP identik** (kelayakan/win_probability/kekuatan/risiko/rekomendasi/asumsi) → renderer frontend tak berubah. Frontend: sumber meta tunggal `client/src/lib/workroom-domains.ts` (WORKROOM_DOMAIN_META + workroomDomainMeta()) → dropdown domain + field dinamis per-domain di `workroom.tsx`, badge domain di list; `workroom-detail.tsx` domain-aware (badge/heading/desc/scoreLabel + ctxSummary generik dari non-textarea fields). CAPSTONE_BY_CATEGORY (`lms-course.tsx`) diperluas: perizinan/legalitas→perizinan, skk/kompetensi→skk, k3/keselamatan→k3 (default tetap tender). Authz workroom TIDAK berubah (getOwnedWorkroom tetap anti-IDOR). Tes authz 36/36 pass, tsc clean, code review PASS.
  - **Why key alignment penting**: key FE meta HARUS sama persis dengan key backend registry (tender/perizinan/skk/k3); drift → domain fallback diam-diam ke tender. Arsitek sarankan (opsional, belum dikerjakan) pindah registry ke modul shared FE+BE.

- **Opsi #2 — Sambungkan lebih banyak kursus Academy ke Workroom** ✅ SELESAI: temuan kunci — 3 kursus "domain" (SBU/SKK/K3, seed `server/routes-lms.ts`) SEMUA `category==="konstruksi"`, hanya beda `subcategory` (SBU/SKK/K3). Mapping capstone lama yg cuma pakai category → SKK & K3 salah arah ke Workroom `tender`. Fix (frontend-only, `lms-course.tsx`): tambah `CAPSTONE_BY_SUBCATEGORY` (SBU→perizinan [via OSS], SKK→skk, K3→k3) + resolver `capstoneForCourse(course)` precedence **subcategory → category → null** (subKey `.trim().toUpperCase()`, catKey `.trim().toLowerCase()`), dipakai di mutation `domain` + render kartu capstone. Kursus onboarding (subcat Dasar/Lanjutan, cat onboarding) → null → tak ada kartu (benar). Tak ada perubahan skema/endpoint. Tes authz 36/36, tsc clean, code review PASS.
  - **Why subcategory-first, bukan category**: kursus satu category bisa memetakan ke banyak Workroom domain; matching by category terlalu kasar. Saat menambah kursus/domain Academy baru, WAJIB petakan lewat subcategory (atau slug) dulu — jangan ulangi bug category-only. Belum ada kursus Perizinan sendiri (SBU dipetakan ke perizinan krn materinya OSS-RBA/kesiapan izin usaha).

- **Opsi #3 — Perkuat Monetization di OS** ✅ SELESAI: tambah fitur **Penghasilan Kreator** read-only yang menutup loop OS (Competency → Monetization), meniru pola Fase 3 Portofolio. Backend `GET /api/earnings` (isAuthenticated, userId-scoped, anti-IDOR — identity dari sesi saja, tak ada target id user-controlled) agregasi dari tabel `store_orders` yg SUDAH ADA (`creator_user_id = userId`) — TANPA tabel baru. Output: stats (paidEarned/pendingEarned per status paid|completed vs pending, count, listedCount agen `is_listed` milik user) + ≤20 pesanan terbaru (itemName via LEFT JOIN store_products/agents). **PII pembeli (customer_name/email/phone) SENGAJA tidak diekspos** — kreator hanya lihat bagi hasilnya. Semua agregat pakai cast `::int` → JS number (hindari bigint string dari SUM). Frontend: `client/src/pages/penghasilan.tsx` (route `/penghasilan`, dark theme, loading/error/empty), kartu di pilar monetization `gustafta-os.tsx`. Tes authz 36/36, code review PASS.
- **Opsi #4 — Poles OS + navigasi** ✅ SELESAI: audit dead-link OS lolos (24 destinasi di `gustafta-os.tsx` semua map ke `<Route>` nyata di `App.tsx`); OS reachable via kartu "Aksi Cepat" `card-gustafta-os` di dashboard. Poles terkontrol (frontend-only, `gustafta-os.tsx` saja, additive/non-breaking): (1) count per pilar `· N tujuan` (`pillar.destinations.length`, auto-sync) di header pilar; (2) footer penutup + tombol "Kembali ke Dashboard" (dulu halaman berhenti mendadak). Tanpa perubahan skema/backend/authz. Tes authz 36/36, code review PASS.
  - **Guard navigasi ke depan**: saat menambah/mengubah destinasi OS, jaga paritas href↔Route — kini OTOMATIS via `tests/gustafta-os-nav.test.ts` (statik: ekstrak `href:"/..."` dari gustafta-os.tsx + `<Route path>` dari App.tsx, cocokkan exact/param, ada min-count floor). shared-header = nav publik/marketing (Beranda/Belajar/Merakit/Store), OS sengaja hidup di alur dashboard (bukan header publik).
  - **Sub-grup dalam pilar**: `Destination.group?` opsional → render bercabang: pilar tanpa group = grid rata (perilaku lama utuh, pilar 01–04); pilar dengan group = tiap grup di bawah subjudul kecil berwarna `iconColor` (urutan grup = urutan array). Dipakai HANYA di Pilar 05 (COMPETENCY & MONETIZATION, 7 kartu 2 konsep) → "Bukti & Kompetensi" vs "Menghasilkan". Aturan: pakai grup HANYA bila pilar benar-benar mencampur >1 konsep, jangan over-segmentasi. Tes authz 36/36, nav test 2/2, code review PASS.

  - **Why fokus marketplace, bukan afiliasi**: `store_orders.creator_user_id` link langsung ke app userId (bersih, anti-IDOR). Afiliasi MLM (`affiliates`/`mlm_commissions`) di-key by affiliateId/email → mapping ke user login rapuh & rawan PII; biarkan lewat link `/affiliate` yg sudah ada. Bila kelak mau earnings afiliasi di OS, cocokkan by email hati-hati (read-only, jangan tulis).

- **Opsi #5 (Mitra Creator / marketplace) DITAHAN — gerbang workshop** ⏸️: keputusan produk pemilik — program Creator TIDAK dibuka dulu; baru dibuka bagi yang sudah ikut **workshop** (verifikasi kemampuan sebelum boleh jual di toko Gustafta). **How to apply**: fitur "Penghasilan Kreator" (Opsi #3) sudah live tapi forward-looking — biarkan tampil (non-creator lihat state kosong), JANGAN promosikan jalur jadi-Creator sebagai self-serve. Saat Creator resmi dibuka pasca-workshop, aktifkan alurnya (mis. gate by flag/workshop-completion), jangan buka pendaftaran Creator bebas.
  - **Prioritas berikutnya = KEMUDAHAN AKSES USER** (pemilik tetapkan sbg utama; detail urgent ditentukan pemilik belakangan). Kandidat yg sudah dibahas: (1) surface halaman baru (Penghasilan/Portofolio) di menu/dashboard sehari-hari, bukan hanya via `/os`; (2) empty-state ramah + CTA di `penghasilan.tsx`; (3) seragamkan tombol "Kembali ke OS" antar-halaman pilar; (4) sambungkan Penghasilan↔Store. Tunggu pilihan pemilik sebelum eksekusi.
