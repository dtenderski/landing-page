# Vision (founder, stated)
Jasa Order sold via the Proposal tool is really about **forming work TEAMS for a client** — e.g. a marketing team, an administration team — several functional teams, member count scaled to complexity. The Gustafta Marketing team (orchestrator + specialists + ◆ human gate) is the reference example.

# Three separate flows today (NOT auto-wired into one pipeline)
1. **Dialog Gustafta** (`/dialog-gustafta`, public) — lead-gen funnel. Staged s1→s1_gate(Profil)→s2→s2_gate(Gambaran+offer)→s3(unlock via Starter Kit)→blueprint TEXT. Gates here = lead-capture/paywall, output is a SINGLE-agent Blueprint text, ends at WhatsApp/Starter Kit. Not connected to the org builder.
2. **Blueprint Builder** (`/blueprint-builder`, auth) — designs ONE agent (Tahap 1–13), `/configure` creates it.
3. **Organization Builder** (`/organization-builder`, auth) — THE team-formation surface. Steps intro→members→review→done. Roles: Ketua Tim(orchestrator)/Spesialis(specialist)/support. Hierarchical (computeOrgTree → tiered teams). **Per-member ◆ Gerbang Manusia (`gates?: string[]`)** citing ebook Buku II Bab 3. AI endpoints: `/api/organization/{suggest,infer,configure,drafts}`. `configure` atomically creates all agents + wires `agenticSubAgents` from `structure.edges`. Runtime delegation already works (proof: Marketing team).

# Ready vs gaps (for the "sell team formation" vision)
- READY: org design (hier. teams), AI-suggest members per domain, AI-write prompts, per-member ◆ gate, atomic create+wire, save drafts, client handover mechanism (agent collaborators + pending invites, viewer/editor).
- ~~GAP — multi-department~~ **DONE (Fase B)**: `suggest` now emits 3-level orgs when the mission spans ≥2 domains (Kepala Kantor → per-domain Ketua Tim → specialists) with default gates.
- ~~GAP — Proposal→org handoff~~ **DONE (Fase C)**: "Rakit Tim Ini" button on Proposal converts `tim_agen` via pure `shared/proposal-to-org.ts` → sessionStorage one-shot handoff (`gustafta_org_prefill_v1`) → org builder consumes on mount (priority over draft restore).
- ~~GAP — Dialog→org handoff~~ **DONE (Fase G+, final roadmap gap)**: public Dialog blueprint (single-agent) → org seed via pure `dialogBlueprintToOrgDraft()` in `shared/proposal-to-org.ts` (one orchestrator from namaChatbot/persona, mission composed from ringkasan+target+persona, all clamped). CTA "Kembangkan Jadi Tim AI" in dialog-gustafta.tsx writes same `gustafta_org_prefill_v1` key + navigates. Tests in `tests/proposal-to-org.test.ts`. **Critical gotcha (guest flow):** org-builder is auth-gated but Dialog is PUBLIC — the mount effect that reads+REMOVES the prefill MUST be gated on `isAuthenticated` (deps `[isAuthenticated, authLoading]`, guarded by bootRef), else a guest click wipes the one-shot prefill before the login gate renders and it's gone after login. sessionStorage survives the login redirect; deferring consumption until authenticated is what makes the handoff seamless.
- ~~GAP — client handover~~ **DONE (Fase D)**: `POST /api/organization/handover` bulk-shares ALL of a team's agentIds to one client email in one action; per-agent `assertCanManageCollaborators` (fail-one-continue, not abort); reuses extracted `shareAgentWithEmailInternal` (single-agent collaborator route now delegates to it too). UI: "Serahkan Tim ke Klien" card in org-builder done step. Anti-probing: unauthorized agents' names redacted from results.
- ~~GAP — default ◆ gates~~ **DONE (Fase E)**: org-builder `suggestTeamComposition` now gives risky-function specialists a curated per-function ◆ gate (advisory/support stay ungated to avoid noise); previously only leads had gates. Gate string carries NO ◆ glyph. IMPORTANT wiring: `applyTeamSuggestion` does NOT copy `gates` — the CLIENT draft (organization-builder.tsx, /suggest → member.gates → gateBlock() into systemPrompt at configure) is the source-of-truth for gates. Don't add gate-carrying to server replay paths expecting applyTeamSuggestion to preserve them.
- ~~GAP — claw-as-management~~ **DONE (Fase F)**: Proposal generator is now "claw-aware" — it can recommend an existing MultiClaw **package (bidang)** as a ready "manajemen AI siap pakai" (AI management department) alongside the custom `tim_agen`. Catalog fed to LLM via `formatClawCatalogForPrompt()` (deterministic from `CLAW_PACKAGES`, package-level). New output field `claw_rekomendasi`.
- ~~GAP — tier→team size/structure~~ **DONE (Fase G)**: deterministic `tierPlanToTimAgen(tier)` + `parseTierNumber(str)` in `shared/team-blueprints.ts`. Proposal route backfills `tim_agen` from the recommended tier's blueprint ONLY when the LLM returns none. **Decision (scope):** do NOT "repair" a non-empty tailored team toward the baseline. **Why:** the baseline is a FLOOR for the empty case, not a target — a non-empty LLM team is already grounded on the blueprint (Fase A) and tailored to the client, so it's the intended product output; and mechanically topping a thin Tier-4 team back up to baseline is impossible to do safely (Tier 4 legitimately has multiple leads — one central coordinator + one per department — so "top up without duplicating a lead" can't reach the headcount, and forcing it risks duplicate coordinators that corrupt org wiring).

# Agreed direction
Founder chose "explain + roadmap first, don't build yet." Roadmap phases: A) Proposal→team bridge (tier→standard team blueprint incl. default gates), B) multi-department in org builder, C) wire Dialog/Proposal→org builder prefill, D) client handover flow, E) default ◆ gates per function.

# Fase A — DONE
Canonical single source `shared/team-blueprints.ts` (TIER_TEAM_PLANS 1-4, `formatTeamPlansForPrompt()`) maps tier→standard team structure + default ◆ gates. Tiers 1-3 = one team; Tier 4 = multi-department (Kepala Kantor lead + Marketing/Administrasi/Operasional). Proposal route grounds the LLM prompt on it; `tim_agen` items gained `tim`+`gerbang`; frontend groups members by team and shows ◆ per member.
**Why/gotchas:** gate strings in the blueprint MUST NOT embed the ◆ char — ◆ is added only at render/prompt time, else the model echoes it and normalization must strip a leading ◆ to avoid "◆ ◆". Any future proposal/org seeding path should reference this module, not re-hardcode team structures (avoid drift). Reuse it when building Fase B/C.

# Fase F — DONE (claw as ready AI management department)
Offering an existing claw to a company via Proposal is a **product-catalog recommendation**, not just collaborator sharing. Two durable rules:
1. **Recommend at PACKAGE (bidang) level, and validate against the catalog.** The LLM must never invent claw names. Server normalizes every `claw_rekomendasi[].nama` through `resolveClawPackageName()` (shared/claw-packages.ts) which maps free text → canonical `CLAW_PACKAGES` name (exact → substring → description-contains, e.g. "SKK K3" → "K3 & Keselamatan") or drops it. Dedup per package. **Why:** GPT tends to return an individual claw name from a package's description rather than the package itself; a name+ID-free allowlist is the only guard against hallucinated/off-catalog offers.
2. **A claw is a PREMIUM LICENSED product — the proposal MUST say so.** Claw recommendation carries its own commercial treatment (lisensi premium + bulanan hosting), separate from the Jasa setup fee. The prompt forces this note into `kenapa_cocok` and the UI section states it. **Why:** without it, a sales draft can imply the claw is bundled/free and misrepresent pricing.
Tests: `tests/claw-packages.test.ts` locks resolver + catalog formatter behavior.
